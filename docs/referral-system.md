# Referral System

## Что делает функционал

В боте реализована реферальная система на базе Telegram deep link:

```text
https://t.me/<bot_username>?start=ref_<referrerUserId>
```

Когда новый пользователь заходит по такой ссылке:

1. Сохраняется, кто его пригласил.
2. Создаётся связь `referrer -> invitedUser`.
3. Моментальный бонус не выдаётся.
4. Бонусы начисляются только после реальной активности приглашённого.

Это снижает риск абуза через фейковые аккаунты и случайные переходы без использования бота.

## Пользовательские команды

- `/ref` — показать персональную реферальную ссылку.
- `/partners` — показать статистику по приглашённым пользователям.
- `/top` — показать топ партнёров.

После успешной генерации изображения бот также показывает кнопку шаринга результата с реферальной ссылкой.

## Логика начислений

### 1. Регистрация реферала

Регистрация происходит на команде `/start ref_<userId>`.

Условия:

- нельзя пригласить самого себя;
- нельзя перезаписать уже существующего `invitedBy`;
- рефералка регистрируется только для нового пользователя;
- для уже существующего пользователя связь не создаётся.

### 2. Активация реферала

Приглашённый считается активированным, если выполнил одно из действий:

- отправил 3 успешных сообщения в чат;
- сгенерировал изображение;
- сгенерировал сцену;
- сделал первую оплату.

После активации:

- приглашённый получает `10_000` текстовых токенов;
- пригласивший не получает токены на этом этапе.

Бонус активации начисляется только один раз.

### 3. Бонусы за оплату

Если приглашённый оплачивает продукт, пригласивший получает бонус процентом.

Текущее правило:

- за каждую оплату приглашённого пригласивший получает `50%` бонусом;
- бонус начисляется только с оплат, не с активации.

Как считается бонус:

- для подписок процент начисляется текстовыми токенами от объёма купленного плана;
- для image pack процент начисляется image credits от купленного пакета.

Примеры:

- если приглашённый купил план на `800_000` токенов, пригласивший получит `400_000` токенов;
- если приглашённый купил пакет на `20` изображений, пригласивший получит `10` image credits.

## Firebase структура

Используются две основные ветки.

### `users/{userId}`

Пример полей:

```text
users/{userId}
  userId: 123456
  referralCode: "123456"
  invitedBy: 987654
  referralActivated: true
  referralPaid: true
  referralRegisteredAt: 1717000000000
  createdAt: 1717000000000
  updatedAt: 1717000000000
```

Назначение:

- хранение кода пользователя для реферальной ссылки;
- хранение обратной связи `invitedBy`;
- быстрые флаги активации и первой оплаты.

### `referrals/{referrerId}/{invitedUserId}`

Пример полей:

```text
referrals/{referrerId}/{invitedUserId}
  referrerId: 987654
  invitedUserId: 123456
  status: "registered" | "activated" | "paid"
  createdAt: 1717000000000
  activatedAt: 1717000010000
  firstPaidAt: 1717000020000
  activityMessages: 3
  earnedTokens: 0
  paidBonusTokens: 400000
  paidBonusImageCredits: 10
  lastPartnerRatePercent: 50
  paidReferralCountAtBonus: 1
  activationSource: "chat_messages" | "image" | "scene_image" | "payment"
```

Назначение:

- хранение состояния конкретной реферальной связи;
- накопление бонусов;
- подсчёт статистики для `/partners` и `/top`.

## Сценарии работы

### Сценарий 1. Новый пользователь по ссылке

1. Пользователь A вызывает `/ref`.
2. Бот показывает ссылку `https://t.me/<bot_username>?start=ref_A`.
3. Пользователь B переходит по ссылке.
4. Бот на `/start ref_A` регистрирует связь `A -> B`.
5. Бонусы пока не выдаются.

### Сценарий 2. Активация сообщениями

1. Пользователь B начинает общение.
2. После 3 успешных сообщений связь переходит в `activated`.
3. Пользователь B получает `10_000` токенов.
4. Пользователь A пока не получает бонус.

### Сценарий 3. Активация генерацией

1. Пользователь B генерирует изображение или сцену.
2. Если активации ещё не было, она срабатывает.
3. Начисляются стандартные бонусы активации.

### Сценарий 4. Оплата приглашённого

1. Пользователь B делает оплату.
2. Бот пополняет баланс B как обычно.
3. Дополнительно рассчитывается партнёрский бонус для A.
4. Бонус начисляется токенами или image credits в зависимости от купленного продукта.

## Ограничения и допущения

- Реферальная связь привязывается только один раз.
- Самореферал запрещён.
- Существующего пользователя нельзя задним числом сделать приглашённым.
- Бонусы начисляются внутренней валютой бота, а не деньгами.
- Пригласивший получает бонусы только после оплат приглашённого.
- Для подписок бонус считается от текстовых токенов плана.
- Для image pack бонус считается от количества купленных изображений.

## Что стоит учитывать при дальнейшем развитии

- Если захочешь показывать красивую историю выплат, можно добавить отдельную ветку `referralEvents`.
- Если понадобится антифрод посильнее, можно учитывать дату регистрации, источник, число сообщений и первую осмысленную генерацию.
- Если захочешь сделать “приглашённый тоже получает бонус за первую оплату”, это лучше добавить отдельным типом события, а не смешивать с активацией.
- Если появится web admin, `/top` и `/partners` можно будет строить уже не по runtime-агрегации, а из заранее посчитанных counters.

## Быстрая проверка после деплоя

1. Открыть `/ref` и убедиться, что ссылка формируется корректно.
2. Перейти новым аккаунтом по ссылке `start=ref_<id>`.
3. Отправить 3 сообщения и проверить начисление бонусов обоим пользователям.
4. Выполнить генерацию изображения и проверить кнопку шаринга.
5. Провести тестовую оплату и проверить бонус пригласившему.
